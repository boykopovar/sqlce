# Формат .sdf (SQL CE 4.0) - обзор

Реверс-инжиниринг бинарного формата .sdf (SQL Server Compact Edition 4.0).
Официальной спеки нет, всё восстановлено из декомпилированного
sqlcese40.dll (движок хранения) плюс эмпирическая проверка на тестовых
файлах.

Эталонная реализация: `sdf_reader.py` (Python, только stdlib, read-only).
DLL-исходники (`sqlcese40_dll.c`, `sqlceme40_dll.c`, `sqlceqp40_dll.c`,
`sqlceca40_dll.c`) - это декомпилированное происхождение всех внутренностей
движка, на которые тут идут ссылки, можно смотреть их напрямую по любому
вопросу, не покрытому здесь.

## Структура документов

- `01-rows.md` - извлечение строк из слотов страницы + физический layout строки
- `02-lob.md` - LOB overflow страницы (ntext/image, которые не влезли в строку)
- `03-column-types.md` - коды типов колонок в каталоге + кодировка значений на диске
- `04-catalog.md` - как ищутся таблицы/колонки через системный каталог (кратко, детали в `catalog/`)
- `05-limitations.md` - известные ограничения + API `sdf_reader.py`

Системный каталог (`__SysObjects`) - отдельная большая тема, вынесена в
`catalog/`:
- `catalog/00-schema.md` - схема каталога, 38 колонок
- `catalog/01-row-layout.md` - физический layout строки каталога
- `catalog/02-classification.md` - классификация строк (TABLE/COLUMN/INDEX/CONSTRAINT), эталонные числа по тестовым файлам
- `catalog/03-status.md` - что подтверждено, что нет

## Базовое устройство страниц

- Размер страницы фиксированный, 4096 байт. Размер файла всегда кратен 4096.
- Байт заголовка `[6]` = тип страницы:
  - `0x40` - heap/data страница. Хранит строки таблиц (и каталога, и
    пользовательских таблиц).
  - `0x50` - LOB overflow страница. Хранит "хвост" слишком большого
    значения ntext/image, которое не влезло в строку целиком.
  - `0x00`, `0x10`, `0x20`, `0x30`, `0x60` - прочие внутренние типы
    страниц (заголовок файла, карты свободного места, b-tree/индексные
    страницы). Этот reader их не разбирает. Поле "число слотов" на них
    (`[20:24]`) - мусор, если пытаться читать его как data-страницу.
  - Тип страницы надо проверять всегда, до применения любой специфичной
    для типа логики парсинга.
- Байт заголовка `[16]` = ObjectId владельца на `0x40`-странице, или (когда
  это надёжно - см. `02-lob.md`) id LOB-потока на `0x50`-странице. Это
  разные пространства имён - `[16]` на `0x50`-странице никак не
  соответствует ObjectId какой-либо таблицы.
- `[20:24]` (u32 LE) - младшие 12 бит = число слотов строк на странице
  (только для `0x40`-страниц).
- `[0:4]` = чексумма (игнорируется). Не исследовано: остаток dword'а
  `[16:20]` за пределами самого байта `[16]`.
