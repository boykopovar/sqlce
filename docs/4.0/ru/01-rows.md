# Извлечение строк и физический layout строки

См. `00-overview.md` для общей картины по страницам.

## 1. Массив слотов / извлечение строк (только для 0x40-страниц)

```
num_slots = u32_le(page, 20) & 0xFFF

for slot_index in range(num_slots):
    slot_off = 4 * (1017 - slot_index)          # растёт назад от байта 4068
    flags_byte = page[slot_off + 27]
    if flags_byte & 0x01:
        continue   # удалённый/свободный слот (0x02 = нормальная занятая строка)

    slot_dword = u32_le(page, slot_off + 24)
    record_offset = slot_dword & 0xFFF
    record_len = ((slot_dword >> 12) & 0xFFF) - 4
    row_bytes = page[record_offset+28 : record_offset+28+record_len]
```

Соответствует `Heap::FastReadRow` в декомпилированном движке. `record_len`
- 12-битное число (максимум 4095) - это длина только фиксированной/
in-page части строки. К тому, насколько логически большим является LOB-
значение внутри этой строки, оно отношения не имеет - размер LOB несётся
отдельно (см. `02-lob.md`). Dword по смещению `record_offset+24` внутри
самой строки (не путать со slot_dword выше) - это указатель продолжения
для строки, чья не-LOB часть растянута на несколько страниц - во всех
виденных образцах всегда 0, не обрабатывается.

Строка каталога (`SysColumn`) может быть физически обрублена на диске,
с байтом флагов слота `0x00` (не нормальный занятый `0x02`, и не бит
удаления `0x01`). Такие строки короче, чем требует фиксированная область
метаданных каталога (offsets 41-46, см. `03-column-types.md`), и их надо
пропускать как невосстановимые фрагменты - если у этой колонки нет
другого, полного вхождения где-либо ещё в файле, она навсегда потеряна
для `table_schema()`.

## 2. Физический layout строки

```
[ внутренний заголовок : 4 байта, смысл не нужен для чтения ]
[ presence bitmap : ceil(общее_число_колонок / 8) байт ]
[ bit-value регион : ceil(число_bool_колонок / 8) байт ]
[ фиксированные колонки : в порядке объявления, bit/var/lob колонки полностью исключены ]
[ var-length директория : 2 байта на каждую переменную колонку (включая ntext/image) --
                          ПОЛНОСТЬЮ ОТСУТСТВУЕТ, если все var-колонки NULL ]
[ var-length данные : подряд идущие байты всех переменных колонок ]
```

### Presence bitmap - флаги NULL

- `ceil(общее_число_колонок/8)` байт, покрывает каждую колонку независимо
  от типа.
- Бит = 1 -> NULL/отсутствует. Бит = 0 -> присутствует. LSB-first: ordinal
  колонки `k` (с 1) -> `byte[(k-1)//8]`, бит `(k-1)%8`.
- NULL-колонка из фиксированного региона (int/datetime/money/float/
  numeric/guid/smallint/tinyint/bigint/binary) всё равно резервирует
  полный `fixed_size` байт в строке, заполненный нулями - НЕ пропускается/
  не опускается.
- NULL var-length колонка всё равно получает запись в директории (2
  байта) - кроме случая, когда все var-length колонки в строке NULL,
  тогда весь регион var-директории (и var-данных) отсутствует в строке
  целиком, точка; строка заканчивается на конце фиксированного региона.
- NULL bit-колонка: её бит в presence bitmap установлен; соответствующий
  бит в bit-value регионе бессмысленный/игнорируется (bit-колонки не
  занимают байт в фиксированном регионе независимо от null-ности).

### Bit-value регион

`ceil(число_bool_колонок/8)` байт. Хранит значения bit-колонок, LSB-first,
в порядке объявления среди только bit-колонок. bit-колонки занимают ноль
байт в регионе фиксированных колонок. Упаковка LSB-first внутри каждого
байта и через границу байта (бит 9 -> байт 1 бит 0, бит 10 -> байт 1 бит 1).

### Var-length директория

Каждая запись - u16 LE:
```
start = word & 0x7FFF     # смещение байта в регионе var-данных
compressed = bool(word & 0x8000)
```
Длина = start следующей записи минус start этой (последняя колонка идёт
до конца строки, точнее до конца in-page части строки - см. `02-lob.md`
насчёт того, что значит "конец строки", когда LOB-значение слишком
большое). Если колонка NULL, её срез по этому правилу пустой (start ==
следующий start, или start == конец var-данных для последней колонки) -
декодировать как NULL напрямую из presence bitmap, а не полагаясь на то,
что длина среза 0, так как пустое не-NULL значение выглядело бы
идентично.

Вся var-length директория (и регион var-данных) отсутствует, если все
var-length колонки в строке NULL - см. "Presence bitmap" выше. Проверять
это до попытки читать байты директории.

`0x8000` - это сжатие Unicode 1-байт-на-символ в SQL CE (только для
nvarchar/nchar), НЕ флаг NULL. Сжатый текст: каждый хранимый байт - это
младший байт code unit'а UTF-16, старший байт подразумевается нулём
(декодируется как Latin-1/ASCII).

### LOB-колонки (ntext, image) - заголовок внутри строки

Та же механика var-директории, что у nvarchar, но извлечённый кусок
начинается с 8-байтного заголовка: `[length:u32 LE][reserved:u32 LE,
всегда 0]`, за которым следует до `length` байт payload'а (UTF-16LE текст
для ntext, сырые байты для image) - но реально присутствует ровно
столько из этих байт, сколько физически влезло на страницу строки.
`length` - это истинный полный размер значения, независимо от того,
сколько его реально присутствует inline.

- Если `length` <= байтам, физически доступным после заголовка на этой
  странице: значение целиком inline, декодировать напрямую.
- Если `length` > доступных inline байт: остаток был записан на LOB
  overflow страницы (`page[6]==0x50`) - см. `02-lob.md`. Overflow
  начинается, как только значение превышает 256 байт (128 символов
  UTF-16 для ntext).

У обычных nvarchar/nchar значений такого заголовка нет, overflow-страницы
для них не поддерживаются.
