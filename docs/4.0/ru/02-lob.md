# LOB overflow страницы (page[6] == 0x50)

Про заголовок значения внутри самой строки (8 байт length+reserved) см.
`01-rows.md`, раздел про LOB-колонки.

- Заголовок `[8:12]` (u32 LE) = порядковый номер записи (sequence
  number), счётчик порядка записи, глобальный по всему файлу, не по
  отдельной цепочке. Строго возрастает вдоль одной цепочки и совпадает с
  возрастающим порядком физических страниц файла внутри этой цепочки.
  Разрывы между соседними seq одной цепочки - нормальное явление
  (страницы других цепочек были перемешаны при записи) и не значат,
  что цепочка сломана.
- Заголовок `[16:24]` (8 байт) кодирует id потока как фиксированный
  повторяющийся паттерн: `[id, 0, id, 0, id, 0, id, 0]`, где `id` - один
  байт, общий для всех страниц цепочки одного LOB-значения. Проверять
  надо именно этот паттерн, не просто байт `[16]` отдельно - на
  некоторых страницах байт `[16]` случайно содержит значение, похожее на
  правдоподобный id, но им не являющееся (см. "Надёжность stream-id"
  ниже).
- Payload: байты `[16:4096]` (4080 байт/страница) - сырые данные
  значения, конкатенируются по цепочке в порядке следования.
  Последняя страница цепочки заполнена лишь частично; остаток забит
  нулями и должен отбрасываться обрезкой конкатенированного буфера до
  значения `length`, объявленного в заголовке внутри строки (см.
  `01-rows.md`), а не по содержимому самой страницы.
- Ни одно проверенное поле внутри строки не указывает на id потока.
  Байты сразу после 8-байтного заголовка внутри строки (когда значение
  переполняется) - это не начало payload'а, это какая-то другая
  структура (повторяющиеся 8-байтные группы значений, похожих на
  page-id/offset), точный layout которой не разрешён.

## Надёжность stream-id

Байт `[16]` (и паттерн `[16:24]` вообще) ненадёжен не на каждой
0x50-странице: сырые байты payload'а сами могут случайно совпасть с
правдоподобными значениями id, особенно для image-значений, чьё
содержимое похоже на счётчик байт или другую структурированную
информацию. Страница считается настоящей stream-id-страницей, только
когда её собственные байты `[16:24]` самостоятельно удовлетворяют
паттерну `[id,0,id,0,id,0,id,0]` - проверка использует только байты этой
страницы, без сравнения между страницами.

Страницы, не удовлетворяющие паттерну, группируются вместо этого по
физической смежности номеров страниц: последовательные номера страниц
образуют одну группу, так как страницы цепочки пишутся как непрерывный
блок на диске (одна логическая цепочка всё же может быть разбита больше
чем на один такой непрерывный блок, если между ними при записи были
перемешаны страницы других цепочек). Отдельные группы объединяются в
одну логическую цепочку только тогда, когда их диапазоны sequence-
номеров сами непрерывны с тем же внутренним шагом, что и у
расширяемой группы - а не просто близки по значению.

## Сборка и сопоставление цепочек

1. Сгруппировать все 0x50-страницы в цепочки: надёжные страницы - по
   stream id (`[16]`, проверенному через паттерн `[16:24]`), ненадёжные
   страницы - по смежности номеров страниц (объединяя через пропуски по
   правилу про шаг выше).
2. Отсортировать страницы каждой цепочки по sequence-номеру.
3. Когда inline-payload строки короче, чем длина, объявленная в её
   8-байтном заголовке, взять первую ещё не использованную цепочку,
   чья ёмкость (`num_pages * 4080`) >= объявленной длины, и прочитать
   payload этой цепочки, обрезанный до этой длины.
4. Пометить цепочку как использованную, чтобы её не переиспользовала
   более поздняя строка.

Это сопоставление по ёмкости может ошибочно назначить не ту цепочку,
если в файле есть два и более переполняющихся LOB-значения очень
похожей суммарной длины без другой различающей информации.
