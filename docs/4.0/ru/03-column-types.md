# Коды типов колонок и кодировка значений

## 4. Коды типов колонок (строка SysColumn каталога, фиксированные offset'ы)

Есть два разных байтовых поля, которые не надо путать:

- offset 23 - сырой байт DBTYPE (константа типа OLE DB). Коллизии между
  типами: int/money оба 0x03; float/numeric оба 0x05; image/ntext оба
  0x64; smallint/tinyint оба 0x02; binary/varbinary оба 0x41.
  Только для отображения, ветвиться по нему нельзя.
- offset 41-42 (u16 LE) - внутренний код типа SQL CE. Авторитетный,
  без коллизий для каждого встреченного типа:

| inttype | SQL-тип           | DBTYPE(23) | Фикс. размер (байт)                                                                              |
|--------:|-------------------|:----------:|--------------------------------------------------------------------------------------------------|
|       0 | tinyint           |    0x02    | 1 (unsigned, 0-255)                                                                              |
|       1 | smallint          |    0x02    | 2 (signed)                                                                                       |
|       3 | int               |    0x03    | 4 (signed)                                                                                       |
|       5 | bigint            |    0x68    | 8 (signed)                                                                                       |
|       8 | nvarchar / nchar  |    0x1F    | var (var-dir + var-data)                                                                         |
|       9 | ntext             |    0x64    | var + 8-байтный LOB-заголовок (+overflow)                                                        |
|      10 | binary(N)         |    0x41    | N, фиксированный - сырые байты сидят прямо в регионе фиксированных колонок, без записи в var-dir |
|      11 | varbinary(N)      |    0x41    | var (var-dir + var-data), сырые байты, без семантики бита сжатия текста                          |
|      12 | image             |    0x64    | var + 8-байтный LOB-заголовок (+overflow)                                                        |
|      13 | datetime          |    0x40    | 8                                                                                                |
|      14 | uniqueidentifier  |    0x65    | 16 (сырые байты GUID)                                                                            |
|      15 | bit               |    0x0B    | 0 (упаковано в bit-регионе)                                                                      |
|      17 | float / real      |    0x05    | 8 (IEEE-754 double)                                                                              |
|      18 | money             |    0x03    | 8                                                                                                |
|      19 | numeric / decimal |    0x05    | 19 (см. ниже)                                                                                    |

Дополнительные подтверждённые поля каталога:
- offset 25 (1 байт) - позиция ordinal, с 1.
- offset 43-44 (u16 LE) - объявленный размер в байтах: 4 для int,
  число_символов×2 для nvarchar(N), 8 для datetime/money/float, 16 для
  uniqueidentifier, 8000 для image/ntext (фиксированный sentinel
  объявленного размера LOB, не связан с реальной длиной значения на
  выполнении), 19 для numeric независимо от объявленной точности,
  1/2/8 для tinyint/smallint/bigint, N для binary(N)/varbinary(N).
- offset 45 / 46 (по 1 байту) - точность / масштаб, имеют смысл только
  для numeric/decimal (inttype 19).

varchar/char в SQL CE 4.0 не существует (только nvarchar/nchar).

## 5. Кодировка значений

- int - i32 LE, знаковое.
- tinyint - 1 байт, unsigned (0-255).
- smallint - i16 LE, знаковое.
- bigint - i64 LE, знаковое.
- binary(N) - N сырых байт, фиксированного размера, сидит прямо в
  регионе фиксированных колонок без записи в var-директории (в отличие
  от varbinary).
- varbinary(N) - var-директория + var-данные, сырые байты, без
  семантики текстового сжатия (соглашение про бит 0x8000 - специфика
  nvarchar/nchar, тут не при чём; записи трактовать как сырые срезы
  байт независимо от этого бита).
- datetime - 8 байт: `[миллисекунды-с-полуночи: u32 LE][дни-с-1900-01-01: u32 LE]`.
  Эпоха 1900-01-01, не 1899-12-30 из OLE Automation.
- money - i64 LE, реальное значение = int64 / 10000.
- float/real - IEEE-754 double, LE.
- numeric/decimal - блок 19 байт: `[precision:u8][scale:u8][sign:u8]
  [нескейленное значение: 16 байт, LE, беззнаковая величина]`. sign==1 ->
  положительное, sign==0 -> отрицательное. Реальное значение =
  нескейленное_значение / 10**scale. Размер блока фиксирован 19 байт
  независимо от объявленной точности. ЗАМЕЧАНИЕ: эталонный декодер
  возвращает Python float для этого типа, что теряет точность для
  больших нескейленных значений (17+ значащих цифр); точное декодирование
  требует хранить нескейленное целое и scale отдельно (например через
  decimal.Decimal), а не делить в float.
- uniqueidentifier - 16 сырых байт, стандартный порядок байт GUID
  .NET/COM (`uuid.UUID(bytes_le=...)` в Python).
- bit - упаковано в bit-регионе, LSB-first среди только bool-колонок.
- nvarchar/nchar - срез var-данных, 1 байт на символ если бит сжатия
  установлен (декодируется как ASCII/Latin-1), иначе UTF-16LE.
- ntext - полный payload (inline, или inline+overflow-собранный, см.
  `02-lob.md`) минус 8-байтный LOB-заголовок, декодируется как nvarchar
  (бит сжатия всё так же применяется к payload'у).
- image - полный payload (inline или собранный) минус 8-байтный
  LOB-заголовок; сырые байты, возвращаются как есть.
