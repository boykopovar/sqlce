# Сканирование системного каталога

Полное описание физического layout'а `__SysObjects` (38-колоночная
схема, offset'ы, правила декодирования presence/bit/var-dir, статус
подтверждённости по полям) вынесено в `catalog/` и не дублируется здесь:

- `catalog/00-schema.md` - схема, 38 колонок
- `catalog/01-row-layout.md` - физический layout строки каталога
- `catalog/02-classification.md` - классификация строк, эталонные числа
- `catalog/03-status.md` - что подтверждено, что нет

Реализация: `catalog_parse/syscatalog.py`, внешняя функция
`decode_syscatalog(path) -> list[dict]`.

Offset'ы полей in-memory объекта `Column` из декомпилированных DLL
(вызовы конструктора `Column::Column`) НЕ совпадают с байтами строки
каталога на диске - это offset'ы метаданных query-процессора, к
физическому layout'у строки отношения не имеют (подтверждено: разные
ветки схемы переиспользуют один и тот же offset, что было бы
противоречием, будь это offset общего on-disk layout'а).

Физический layout строки `__SysObjects` (presence bitmap, bit-регион,
58-байтный union фиксированного региона across все четыре вида строк,
20-байтная var-директория) реверс-инженерен полностью и реализован в
`catalog_parse/syscatalog.py`. Детали по каждому полю - в `catalog/`.

## Что sdf_reader.py реально берёт из decode_syscatalog

(см. `SdfFile._scan_catalog` в коде)

1. Страницы каталога ищутся так же, как раньше: сканируются все
   0x40-страницы, берётся `owner_object_id` любой страницы, содержащей
   литерал `__Sys`; каждая страница с этим же ObjectId - страница
   каталога, даже если сама литерала `__Sys` не содержит (строки колонок
   одной таблицы могут переливаться на дополнительные страницы без
   единого текста `__Sys` на них) - `find_catalog_object_ids` /
   `iter_catalog_rows` в `sdf_pages.py`.
2. `decode_syscatalog` декодирует каждую занятую строку каталога
   структурно (без эвристик по ASCII-строкам) и помечает её `_row_kind`
   = TABLE / COLUMN / INDEX / CONSTRAINT.
3. У строки TABLE `SysObjectName` - имя таблицы; младший байт
   `SysTablePageId` - ObjectId таблицы, ключ, связывающий строки
   каталога с физическими страницами данных таблицы (подтверждено на
   всех 4 тестовых файлах: совпадает с `page.owner_object_id`
   собственных страниц данных этой таблицы).
4. Строка COLUMN принадлежит таблице, названной в её `SysObjectOwner`;
   `SysObjectName`/`SysObjectCedbInfo_ordinal`/`_dbtype`/`SysColumnType`/
   `SysColumnSize`/`SysColumnPrecision`/`SysColumnScale` мапятся прямо
   на поля `ColumnDef`. Строки INDEX/CONSTRAINT игнорируются - страницы
   данных ищутся через ObjectId (шаг 3), а не через поле PageId на
   строке индекса, так что строки SysIndex не нужны для чтения таблиц/
   строк.
5. Поиск страниц данных каждой таблицы: сканируется каждая страница
   файла; для каждой 0x40-страницы, чей `owner_object_id` НЕ является
   ObjectId'ом каталога (шаг 1), она приписывается той таблице, чей
   ObjectId (шаг 3) совпадает. Это естественно покрывает
   многостраничные таблицы, так как каждая страница сама объявляет
   своего владельца.

Не реверс-инженерено / вне рамок (актуальный статус по полям - в
`catalog/03-status.md`): поля SysIndex/SysConstraint и большинство
bit-полей SysObject/SysTable кроме Fixed/Nullable; структура
SysIndex/b-tree (строки находятся полным сканированием heap'а, а не
через индексный поиск); структура внутри строки, которая, вероятно,
указывает на id LOB-потока (`02-lob.md`); причина физически обрубленных
строк каталога (`01-rows.md`).
